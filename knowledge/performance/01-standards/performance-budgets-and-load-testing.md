# 性能预算与负载测试规范（商业级必读）

> "感觉挺快"不是性能标准。商业级团队把性能写成**预算**——每条关键路径有明确的延迟/资源上限，超了就当成 bug，在上线前就拦住；并用**负载测试**在真实压力下证明系统在容量内仍守得住这些预算。
> 性能不是上线后被用户骂了才优化的事，而是**事先定额、持续度量、纳入门禁**的工程纪律。

## 1. 核心原则

- **预算先行**：先为关键路径定下"最慢可以多慢、最多用多少资源"，再实现，而不是事后测一下凑合。
- **看分位数，不看平均**：平均值会骗人。盯 **p95 / p99**——尾延迟才是真实用户的痛点。
- **预算即门禁**：性能预算接进 CI / 发布门禁，回归就拦，别等用户发现。
- **在压力下验证**：单用户快不算数，要在**目标并发**下仍满足预算才算达标。
- **留余量（headroom）**：系统要在峰值之上留容量缓冲，不能跑在 100% 上等着被一个尖峰打挂。

## 2. 性能预算定什么

为每条**关键路径**（核心 API、关键页面、关键作业）定可度量的上限：

| 维度 | 预算示例 | 度量 |
|---|---|---|
| 后端接口延迟 | 关键读接口 p95 < 200ms、p99 < 500ms | 服务端指标 |
| 端到端延迟 | 核心操作端到端 p95 < 800ms | 真实用户监控（RUM）|
| 前端加载 | 关键页面在 4G、中端设备上达成核心网页指标的"良好"档 | 实验室 + 现场 RUM |
| 资源体积 | 关键页面首屏 JS ≤ N KB、总传输 ≤ M KB | 构建产物分析 |
| 吞吐 | 系统在目标 QPS 下错误率 < 0.1% | 负载测试 |
| 资源占用 | 单实例峰值 CPU < 70%、内存不增长（无泄漏）| 资源监控 |

要点：预算要写成**带数字 + 带分位 + 带条件**（设备/网络/并发）的可验证目标，不写"要快"（见 `experts/product-manager/requirements-engineering-ears` 对 NFR 的要求）。

## 3. 前端：核心网页指标与资源预算

- **核心网页指标**：加载速度、交互响应、视觉稳定性三类用户体验指标，对每条关键页面设"良好"阈值，并在**现场真实用户数据**（而非只在实验室）上达标——实验室绿、现场红等于没达标。
- **资源预算**：限制关键页面的 JS/CSS/图片体积与请求数；构建时检查，超预算就拦 PR，防止"每次加一点点"温水煮青蛙式劣化。
- **关键路径优先**：预算优先盯首屏与核心交互路径，而不是平均所有页面。

## 4. 负载测试的四种形态（别只跑一种）

| 类型 | 问什么 | 做法 |
|---|---|---|
| 负载（load）| 预期峰值下达标吗？ | 加压到目标峰值并维持，看延迟/错误率是否守预算 |
| 压力（stress）| 极限在哪、怎么坏的？ | 持续加压直到崩溃，找拐点与失效模式（是优雅降级还是雪崩）|
| 尖峰（spike）| 流量突增扛得住吗？ | 瞬间拉高再回落，看能否快速吸收与恢复 |
| 浸泡（soak）| 长时间稳定吗？ | 中等负载跑数小时~数天，抓内存泄漏、资源耗尽、连接池枯竭、缓慢劣化 |

只跑短时负载测试会**漏掉浸泡才暴露的泄漏类问题**——很多生产事故是"跑了三天才挂"。四种各问不同的问题，按风险选跑。

## 5. 测试方法纪律

- **环境贴近生产**：在与生产相近规模/配置的环境压测；玩具环境的数字不可外推。
- **真实负载模型**：用接近真实的请求分布、数据规模、并发曲线，别只压一个最快的接口。
- **数据规模真实**：在生产量级的数据上测（大表的查询才会慢），用合成/脱敏数据（见 `testing/01-standards/test-data-and-ephemeral-environments`）。
- **隔离变量**：一次只改一个因素，否则分不清是谁的影响。
- **建立基线 + 防回归**：记录基线，关键发布前重压，对比是否劣化；理想情况接入流水线定期回归。

## 6. 容量与余量

- **按峰值 + 余量规划**：容量按业务峰值再留缓冲（不跑在满载），应对突发与故障转移后的负载集中。
- **找瓶颈**：通过压测定位最先饱和的资源（CPU/内存/IO/连接池/下游依赖），那就是容量上限所在。
- **自动伸缩有上限与冷启动成本**：弹性扩容不是万能——扩容有延迟、有上限、冷启动有代价，尖峰场景要预热或预留。
- **与韧性联动**：过载时的行为（限流、卸载、降级）见 `backend/01-standards/resilience-and-fault-tolerance`；容量规划深做见 `operations/capacity-planning`。

## 7. 反模式（出现即不合格）

1. **只看平均延迟**：平均很美，p99 很惨，真实用户在尾部受苦。
2. **无预算**：没有事先定额，性能好坏全凭感觉，劣化无人察觉。
3. **只测单用户**：本地一个人点着很快，目标并发下直接崩。
4. **不跑浸泡**：短测全绿，上线三天后内存泄漏/连接耗尽挂掉。
5. **玩具环境压测**：在远小于生产的环境压出的数字拿去承诺生产容量。
6. **空库压测**：在没数据的库上测查询，生产大表上全慢。
7. **跑满不留余量**：容量卡在峰值，一个尖峰或一次故障转移就雪崩。
8. **性能不进门禁**：性能回归靠用户投诉发现，而不是 CI 拦住。

## 8. 最低交付 checklist

- [ ] 关键路径有明确性能预算（带数字、分位、条件），写成可验证目标。
- [ ] 盯 p95/p99 尾延迟，不以平均值判定达标。
- [ ] 前端关键页面对核心网页指标在现场真实数据上达标，并有资源体积预算。
- [ ] 性能预算接入 CI / 发布门禁，回归即拦。
- [ ] 在贴近生产的环境、真实负载模型与数据规模下做负载测试。
- [ ] 按风险覆盖负载/压力/尖峰/浸泡，浸泡测试抓泄漏类问题。
- [ ] 容量按峰值 + 余量规划，已定位主要瓶颈资源。
- [ ] 建立性能基线，关键发布前重压对比、防回归。
