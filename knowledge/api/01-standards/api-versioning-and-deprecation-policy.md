# API 版本与弃用治理规范（商业级必读）

> 接口一旦有人调用，它就是**契约**，不再是你一个人能随便改的代码。
> 商业级团队对外接口的演进遵循一条铁律：**老调用方不许被你的改动搞挂**。
> 版本化解决"如何在不破坏现有消费方的前提下演进"，弃用治理解决"如何**有序、可预期地**让老版本退场"——两者缺一，要么永远不敢改，要么一改就出事故。

## 1. 核心原则

- **兼容优先**：能向后兼容就别升版本。新增是兼容的，删改是破坏的。
- **破坏性变更必须版本化**：永远不在同一版本里做破坏性变更（见 §3 判定）。
- **弃用要可预期**：老版本退场要提前公告、给迁移期、给信号，而不是某天突然 404。
- **消费方视角**：以"现有调用方会不会挂"为唯一判据，而不是"我代码改得爽不爽"。
- **支持窗口有限且明确**：不无限维护所有历史版本，但退场节奏要写明、要守约。

## 2. 版本化方式（选一种，全项目统一）

| 方式 | 形态 | 优点 | 取舍 |
|---|---|---|---|
| URL 路径 | `/api/v1/orders` | 直观、易路由、缓存友好 | 版本粒度粗（整套接口）|
| 请求头 | `API-Version: 2` | URL 稳定、灵活 | 不直观、调试需看 header |
| 日期版本 | `API-Version: 2026-06-29` | 渐进、精确到每次变更 | 管理成本高 |
| 媒体类型 | `Accept: application/vnd.x.v2+json` | 符合 HTTP 语义 | 客户端支持参差 |

要点：**对外公开 API 推荐 URL 路径版本**（最直观、最少踩坑）；选定后全项目统一，别混用。主版本号对应破坏性变更（语义化版本，见 `release-engineering/01-standards/progressive-delivery-and-release`）。

## 3. 破坏性 vs 兼容性变更（判定表）

| 破坏性（必须升主版本）| 兼容性（不升版本）|
|---|---|
| 删除字段 / 端点 | 新增**可选**字段 / 新端点 |
| 重命名字段 | 新增可选请求参数（有安全默认）|
| 改字段类型 / 取值含义 | 放宽校验（接受更多输入）|
| 把可选参数改为必填 | 新增枚举值（**需消费方容忍未知值**）|
| 改默认行为 / 改错误码语义 | 新增响应头 |
| 收紧校验（拒绝原本接受的输入）| bug 修复（不改契约语义）|
| 改认证 / 分页 / 排序契约 | 性能优化（行为不变）|

判定底线：**任何让现有调用方的正确请求开始失败、或让它依赖的返回结构变形的改动，都是破坏性的。** 新增枚举值是否破坏，取决于你是否在契约里要求消费方"忽略未知值"——要求了才算兼容。

## 4. 弃用生命周期（三阶段，可预期）

```
公告(announce) → 过渡(transition) → 下线(sunset)
```

| 阶段 | 动作 |
|---|---|
| 公告 | 在文档/changelog 标记弃用，给出替代方案、迁移指南、**明确下线日期** |
| 过渡 | 老版本继续可用但返回弃用信号（见 §5），主动联系高频调用方，监控剩余用量 |
| 下线 | 用量降到可接受阈值且过了承诺期后，停用老版本并返回明确错误 |

铁律：**先有替代、再宣弃用**——没有可迁移的新版本，不准弃用老版本。给足真实的迁移窗口（按受众，常以季度/年计），不是说下线就下线。

## 5. 弃用信号（让调用方提前知道）

- **`Deprecation` 头**：标记该端点/版本已弃用（可带弃用时间）。
- **`Sunset` 头**：标明计划下线时间，让调用方有据可依地排期迁移。
- **`Link` 头 / 文档**：指向替代端点与迁移指南。
- **可观测**：服务端监控老版本调用量与调用方，弃用进度数据驱动，而不是猜。
- **changelog**：每次版本变更面向使用者写清"新增/变更/弃用/破坏性/迁移方式"。

## 6. 治理与协同

- **契约即真相**：接口契约（OpenAPI 等）是版本化和兼容性校验的依据；契约变更要过契约测试（见 `testing/01-standards/contract-testing-and-api-contracts`）。
- **兼容性门禁**：CI 中校验新契约对现有消费方兼容（无未升版的破坏性变更），不兼容则拦住发布（见 `testing/01-standards/ci-test-gates-and-coverage`）。
- **消费方清单**：维护"谁在调哪个版本"，弃用前能精准触达，下线前能确认无活跃调用。
- **内部 vs 外部**：内部接口可更快迭代、更短窗口；公开/合作方接口要更长支持期与更正式公告。
- **与开关的边界**：功能开关控行为可见性，**不替代**版本化（见 `release-engineering/01-standards/feature-flag-lifecycle`）。

## 7. 反模式（出现即不合格）

1. **破坏性变更不升版**：直接改线上契约，老调用方批量 500/解析失败。
2. **静默弃用 / 突然下线**：不公告、不给迁移期，某天直接 404。
3. **无替代就弃用**：宣布老版本退场却没有可迁移的新版本。
4. **无限维护所有版本**：从不下线，版本越堆越多，维护成本失控。
5. **版本方式混用**：URL、header、query 各处混搭，调用方无所适从。
6. **不知道谁在调**：没有消费方与用量数据，不敢动也不敢下线。
7. **把收紧校验当兼容**：以为"只是更严格"，其实让原本合法的请求开始被拒。
8. **changelog 堆 commit**：使用者看不懂变了什么、要不要迁移。

## 8. 最低交付 checklist

- [ ] 选定一种版本化方式并全项目统一；破坏性变更升主版本。
- [ ] 明确区分破坏性/兼容性变更，破坏性绝不进同版本。
- [ ] 弃用走公告→过渡→下线三阶段，**先有替代再弃用**，给足迁移窗口。
- [ ] 弃用时返回 `Deprecation`/`Sunset` 信号，文档给出迁移指南。
- [ ] 监控各版本调用量与消费方，弃用/下线数据驱动、可精准触达。
- [ ] 契约为准，CI 做兼容性门禁，破坏性变更被拦截。
- [ ] changelog 面向使用者写清变更与迁移方式。
- [ ] 明确版本支持窗口，到期有序下线、守约不突袭。
