# 可观测性与 SLO 运营规范（商业级必读）

> 监控回答"系统是不是挂了"，可观测性回答"**为什么挂、挂在哪、影响谁**"——尤其是面对**从没预想过**的故障，你能不能只靠已有的遥测数据查出来，而不必先发一版新代码加日志。
> 商业级系统把"每个服务必须发出什么遥测"当成**埋点契约**来落地，把三类信号（日志/指标/追踪）用统一关联键打通，并用 **SLO + 错误预算**把"健康"从主观感受变成可运营的数字。

## 1. 三类信号，各司其职

| 信号 | 回答 | 特点 |
|---|---|---|
| 指标（metrics）| "坏了吗？坏多少？"（趋势、聚合）| 便宜、可长留、适合告警与看板 |
| 日志（logs）| "具体发生了什么？"（离散事件细节）| 信息密度高、量大、要结构化 |
| 追踪（traces）| "慢/错在调用链的哪一跳？"（请求穿越多服务的路径）| 定位跨服务瓶颈与故障点 |

三者不是三选一，而是**互补**：指标发现异常 → 追踪定位到哪一跳 → 日志看那一跳的细节。关键是它们能**互相关联**（见 §3）。

## 2. 埋点契约（每个服务必须发出的最小遥测）

把"该发什么"标准化，避免每个服务各记各的、查问题时拼不起来：

- **结构化日志**：JSON/键值，不是裸文本字符串。必带字段：时间戳（UTC、ISO-8601）、级别、服务名、**关联ID（request_id / trace_id）**、消息；事件相关的关键业务字段。
- **请求/依赖指标**：每个入站端点与每个出站依赖发出请求量、错误率、延迟分布（直方图，能算分位）；资源侧发出利用率与饱和度。
- **追踪**：入口生成/承接 trace，向所有下游**传播追踪上下文**，每个关键操作开 span，记录耗时与状态。
- **业务指标**：注册、下单、支付、转化等业务事件计数——技术全绿但下单为零也是事故。
- **健康检查**：存活/就绪端点，区分"进程活着"和"能正常服务"（依赖就绪）。

## 3. 关联：把三类信号缝起来

- **统一关联ID**：每个请求在入口生成 `request_id` / `trace_id`，**贯穿**该请求触发的所有日志、span、跨服务调用。出问题时凭一个 ID 就能把一次请求的日志、链路、指标全捞出来。
- **上下文传播**：调用下游时透传追踪上下文（标准追踪头），让链路不断裂。
- **日志带 trace_id**：每条日志都带当前 trace_id，实现"从一条慢链路一键跳到对应日志"。
- **统一维度**：服务名、环境、版本、端点等标签在三类信号里取**一致的命名**，才能交叉关联。

## 4. 基数（cardinality）纪律

可观测性最容易失控、最容易烧钱的地方是**高基数标签**：

- **指标标签不放无界值**：用户ID、请求ID、邮箱、完整 URL 等高基数值**绝不**做指标标签（会让时间序列爆炸、成本失控、查询变慢）。这些放**日志/追踪**里（它们按事件存，能承载高基数）。
- **路径要模板化**：`/orders/123` 在指标里归一为 `/orders/{id}`，不要每个 id 一条序列。
- **采样**：高流量追踪可采样（按比例 + 对错误/慢请求强制保留），平衡成本与可见性。
- **日志分级与量控**：DEBUG 默认关；避免热路径里高频打无用日志把信号淹没、把成本拉高。

## 5. SLI / SLO / 错误预算（把"健康"变成可运营的数字）

- **SLI（指标）**：从用户视角度量服务质量的量——可用性（成功请求占比）、延迟（达标请求占比）、正确性等。盯**用户体验到的**，不是 CPU。
- **SLO（目标）**：给 SLI 定目标，如"30 天内 99.9% 的请求成功且 p95 < 300ms"。SLO 要现实、可达、对齐业务。
- **错误预算（error budget）**：100% 减去 SLO = 允许的"不达标额度"。99.9% 即每月约 0.1% 的预算。
  - 预算**没烧光** → 可以快速发布、大胆迭代。
  - 预算**烧光** → 冻结高风险变更、优先稳定性，直到回到预算内。
  - 这把"该求快还是求稳"从拍脑袋变成**数据驱动**的决策（深做见 `operations/slo-sli-playbook`）。

## 6. 告警：少而准，可执行

- **基于症状告警**：对**用户感受到的**问题告警（错误率、延迟、SLO 燃尽），而不是对每个内部抖动（单台 CPU 高）告警。
- **错误预算燃尽率告警**：按预算消耗速度告警——烧得快（短窗口高燃尽）才需立刻叫人，慢烧给工单即可。
- **每条告警可执行**：触发即有人能做事，并指向 runbook；不可行动的告警要删，否则养出"告警疲劳"、真事故被淹。
- **分级**：区分"立刻起床处理"和"工作时间看"，别什么都 P1。

## 7. 反模式（出现即不合格）

1. **裸文本日志**：非结构化字符串，无法检索、聚合、关联。
2. **三信号各自为政**：日志、指标、追踪没有共同关联ID，出事拼不起来。
3. **高基数炸指标**：把用户ID/请求ID/原始URL塞进指标标签，序列爆炸、成本失控。
4. **只测技术指标不看业务**：CPU/内存全绿，但下单为零无人发现。
5. **只有指标没有追踪**：知道"变慢了"但不知道慢在跨服务调用的哪一跳。
6. **告警噪音**：成百上千条不可行动告警，真事故被淹，on-call 疲劳。
7. **无 SLO**：没有"多好才算好"的定义，求快还是求稳全靠吵。
8. **要查问题先加日志再发版**：遥测不足，未预想的故障无法靠现有数据定位。

## 8. 最低交付 checklist

- [ ] 全部日志结构化，带时间戳/级别/服务名/关联ID/消息。
- [ ] 每个端点与依赖发出请求量/错误率/延迟分布；有关键业务指标。
- [ ] 入口生成 trace_id 并贯穿日志与跨服务调用，追踪上下文全程传播。
- [ ] 指标标签无高基数值，路径模板化；高流量追踪按需采样、错误强留。
- [ ] 定义面向用户的 SLI 与现实的 SLO，落地错误预算并用于发布决策。
- [ ] 告警基于用户症状与错误预算燃尽，条条可执行、指向 runbook、分级。
- [ ] 健康检查区分存活与就绪。
- [ ] 凭一个 request_id 能捞出一次请求的日志、链路与相关指标。
