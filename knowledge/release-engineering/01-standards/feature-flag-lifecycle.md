# 功能开关全生命周期规范（商业级必读）

> 功能开关是把"部署"和"发布"解耦的核心工具：代码先上、功能后开、出事即关。
> 但开关用不好会变成最隐蔽的技术债——代码里堆满永不删除的死分支、谁也不敢动的"祖传开关"、测试矩阵指数爆炸。
> 商业级团队把开关当**有生命周期、要被回收**的资产管理，而不是随手 `if` 一下。

## 1. 开关的四种类型（寿命与纪律不同）

| 类型 | 用途 | 寿命 | 纪律 |
|---|---|---|---|
| 发布开关（release）| 新功能藏在后面先部署，再灰度打开 | **临时**，上线稳定后清理 | 必须设清理期限，否则成债 |
| 实验开关（experiment / A-B）| 按人群分流做对比实验 | 临时，实验结束即清理 | 实验有明确结束条件 |
| 运营开关（ops / kill-switch）| 应急熔断、降级、限流 | **长期**保留 | 默认安全态，变更要审计 |
| 权限开关（permission）| 按租户/套餐/地域控制可见性 | 长期，随业务存在 | 与计费/权限系统对齐 |

**先分类再创建**：一个开关是"临时发布开关"还是"长期运营开关"决定了它要不要被清理。把临时开关当长期用 = 开关债的源头。

## 2. 生命周期五阶段

```
创建 → 灰度放量 → 全量稳定 → 清理 → 归档
```

| 阶段 | 关键动作 |
|---|---|
| 创建 | 登记开关（名称、类型、负责人、目的、预期清理日期）；默认值设为**安全态** |
| 灰度放量 | 按比例/人群/地域逐步打开（1%→5%→25%→100%），每步看指标 |
| 全量稳定 | 100% 打开并观察足够时间，确认无回退需求 |
| 清理 | 删除开关判断 + 旧分支代码，只留新路径 |
| 归档 | 记录开关曾存在（审计/复盘可查），从活跃清单移除 |

临时开关的**清理是交付的一部分**，不是"以后有空再说"——开关进了代码，对应的清理任务就要进待办。

## 3. 创建规范

- **登记**：每个开关有名称、类型、负责人、目的、创建日期、**预期清理日期**（临时开关必填）。
- **命名清晰**：从名字能看出它控什么、是临时还是长期（如 `release_new_checkout`、`ops_payment_killswitch`）。
- **默认安全**：默认值是"出问题也安全"的那个态——新功能默认关、熔断默认不触发。
- **单一职责**：一个开关控一件事，别让一个开关同时管三个无关功能。

## 4. 灰度与定向

- **维度**：按百分比、用户ID哈希、租户、套餐、地域、内部员工等定向放量。
- **稳定分桶**：同一用户每次落同一桶（基于稳定哈希），避免功能"忽闪忽现"破坏体验。
- **逐步放量 + 看板**：每放量一档看错误率/延迟/核心指标，异常即回退（关开关，无需发版）。
- **应急可关**：任何放量中的开关都能一键关回安全态，这是开关相对"重新发版回滚"的核心价值。

## 5. 与代码的纪律

- **开关判断集中**：开关读取走统一接口，不在代码各处散落硬编码字符串；便于盘点和清理。
- **避免嵌套开关**：多个开关嵌套组合 → 测试路径指数爆炸、行为不可预测。尽量扁平、互不依赖。
- **两条路径都要能跑**：开关开/关两个分支都要被测试覆盖（至少冒烟），别只测打开态。
- **开关不替代版本兼容**：开关控制的是"行为是否可见"，破坏性的对外接口变更仍要走版本化（见 `release-engineering/01-standards/progressive-delivery-and-release`、`api/01-standards/api-versioning-and-deprecation-policy`）。

## 6. 清理（对抗开关债）

- **到期回收**：临时开关到预期清理日仍未清，触发提醒/拦截；把"过期开关数"当技术债指标盯。
- **清理即删码**：删开关不是把它常开，而是**删掉判断和已废弃的旧分支**，让代码只剩胜出的那条路径。
- **定期盘点**：周期性扫描所有开关，标记"长期 100% 打开的临时开关"——它们就是该清理的死分支。
- **审计长期开关**：运营/权限类长期开关的变更（谁、何时、改成什么）要留痕，尤其 kill-switch。

## 7. 反模式（出现即不合格）

1. **开关债**：临时开关上线后从不清理，代码堆满永久死分支。
2. **僵尸开关**：长期 100% 打开却没人敢删，因为不知道关掉会怎样。
3. **默认不安全**：默认开 / 默认放量，新代码一上即全量暴露，失去"先部署后发布"的意义。
4. **嵌套开关地狱**：开关层层嵌套，组合爆炸，没人能推断真实行为。
5. **散落硬编码**：开关名字符串散落各处，无法盘点、无法统一清理。
6. **只测打开态**：关闭分支从不测，回退时才发现旧路径早已不工作。
7. **无登记无负责人**：开关没人负责、没清理日期，成无主孤儿。
8. **kill-switch 无审计**：应急开关被改了也查不到谁改的。

## 8. 最低交付 checklist

- [ ] 每个开关先分类（发布/实验/运营/权限），临时开关设预期清理日期。
- [ ] 开关已登记：名称、类型、负责人、目的、默认安全态。
- [ ] 灰度按稳定分桶逐步放量，每步看指标，可一键关回安全态。
- [ ] 开关读取走统一接口，不散落硬编码，避免嵌套。
- [ ] 开关开/关两条分支都有测试覆盖。
- [ ] 临时开关上线稳定后及时清理（删判断 + 删旧分支），不留死分支。
- [ ] 定期盘点开关，过期/僵尸开关纳入技术债跟踪。
- [ ] 长期运营/权限开关变更留审计，kill-switch 默认安全态。
