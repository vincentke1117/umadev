# 渐进式发布与版本交付规范（商业级必读）

> "全量上线、出事再说"是业余做法。商业级发布把风险**逐步释放、随时可回退**：
> 先小流量验证、再放量、出问题秒回滚。发布是一次可控、可观测、可逆的工程动作，不是一次祈祷。

## 1. 发布原则

- **可逆优先**：任何发布都要有明确、已验证的**回滚路径**。不可逆的变更（数据迁移、删字段）单独走、提前演练。
- **小步快跑**：小批量、高频次发布，比大版本一次性上线风险低、定位快。
- **渐进释放**：新版本流量从 0 逐步放大（金丝雀→放量→全量），每一步看指标再决定继续还是回退。
- **可观测**：每次发布都能从监控看出"是否变坏"——错误率、延迟、关键业务指标必须在发布前就接好。
- **解耦部署与发布**：先把代码**部署**上去（对用户不可见），再用开关**发布**（对用户可见）——两者分离让上线更安全、回退更快。

## 2. 版本与变更记录

- **语义化版本（主.次.修订）**：破坏性变更升**主**版本，向后兼容的新功能升**次**版本，修复升**修订**版本。对外 API 的破坏性变更必须升主版本（见 `experts/architect/contract-first-api-design`）。
- **变更日志（changelog）**：每个版本记录"新增/变更/修复/破坏性/弃用"，面向使用者写，而非堆 commit。
- **不可变制品**：构建一次产出带版本号的不可变制品，各环境晋级用**同一个**制品（dev→staging→prod 不重新构建），杜绝"环境间不一致"。
- **可追溯**：制品能追到 commit、构建记录、依赖清单，便于事故定位与合规取证。

## 3. 渐进式发布策略

| 策略 | 做法 | 适用 | 回滚 |
|---|---|---|---|
| 金丝雀（Canary）| 新版本先接 1%→5%→25%→100% 流量，逐步放量 | 大多数无状态服务 | 把流量切回旧版本 |
| 蓝绿（Blue-Green）| 旧(蓝)/新(绿)两套并存，流量整体切到绿，旧的留作回退 | 需快速整体切换/回退 | 流量切回蓝 |
| 滚动（Rolling）| 实例分批替换为新版本 | 资源受限、可容忍混跑 | 暂停并回滚批次 |
| 影子（Shadow）| 把真实流量复制给新版本但不返回给用户 | 验证性能/正确性、零用户风险 | 直接停影子 |

每一步都设**晋级判据**（错误率、延迟、业务指标在阈值内）和**自动回退条件**（超阈值自动回滚/暂停），别靠人盯。

## 4. 功能开关（Feature Flag）

- **部署 ≠ 发布**：新功能藏在开关后先部署，灰度时按用户/比例/地域逐步打开，出问题**关开关即回退**（无需重新发版）。
- 开关类型：发布开关（临时，上线后清理）、实验开关（A/B）、运营开关（长期，应急熔断）、权限开关（按租户/套餐）。
- **纪律**：临时开关上线稳定后**及时清理**，否则代码里堆满死分支（开关债）；开关默认值要安全（默认关）；关键开关变更要审计。
- 开关与契约：开关控制的是行为可见性，不替代版本兼容——破坏性接口变更仍需版本化。

## 5. 数据库变更与发布解耦

- 变更要**向后兼容、可分步**：先加（可空列/新表）→ 双写/回填 → 切读 → 清理旧结构。**绝不**在一次发布里"改表结构 + 改代码"强耦合。
- **扩展-收缩（expand-contract）**：先扩展兼容新旧两版代码的 schema，部署新代码，确认稳定后再收缩删旧结构。这样应用回滚时 schema 仍兼容。
- 大回填/迁移与发布分离、可中断、可重入、限速，避免锁表拖垮线上。
- 数据变更前**备份/快照**，并明确"能否回滚数据"——很多数据变更不可逆，要在发布计划里标红。

## 6. 回滚

- 回滚是**一等公民**：每个发布计划必须写明回滚步骤，并在演练/staging 验证过。
- **快速止血优先**：出事先回滚/关开关恢复用户，再慢慢查根因——别在生产上边查边修。
- 回滚要考虑"前向不兼容"：若新版本已写入旧版本读不了的数据，回滚会出错——这正是要求 schema 扩展-收缩、消息向后兼容的原因。
- 定义自动回滚触发：关键指标超阈值，自动回退当前批次并告警。

## 7. 发布门禁（can-i-deploy）

发布前必须全绿（参见 `testing/01-standards/ci-test-gates-and-coverage`）：
- [ ] 所有必过 CI 检查通过（测试/契约/构建/安全扫描）。
- [ ] 契约兼容当前线上所有消费方（无未升版的破坏性变更）。
- [ ] 监控/告警/日志已接好，发布后能立刻看出好坏。
- [ ] 回滚步骤已写明并验证。
- [ ] 数据库变更向后兼容、可回滚或已评估不可逆风险。
- [ ] 高风险时段/变更冻结窗口已规避。
- [ ] 发布负责人 + 回退负责人明确，发布有记录（谁、何时、什么版本）。

## 8. 反模式（出现即不合格）

1. **全量直发**：100% 流量一把切，无灰度无观测。
2. **无回滚预案**：出事现想办法，止血慢。
3. **部署即发布、强耦合**：代码与不兼容的 schema 一次性上，回滚就坏数据。
4. **开关债**：临时开关从不清理，代码堆满死分支。
5. **环境间重新构建**：staging 和 prod 跑的不是同一个制品。
6. **发布无观测**：上完线靠用户投诉才知道坏了。
7. **不可逆变更不预警**：删字段/数据迁移混在普通发布里。

## 9. 最低交付 checklist

- [ ] 语义化版本 + 面向用户的 changelog；制品不可变、跨环境晋级同一份。
- [ ] 采用金丝雀/蓝绿等渐进式策略，按指标分步放量。
- [ ] 用功能开关解耦部署与发布，开关可应急关闭且会清理。
- [ ] 数据库变更走扩展-收缩、向后兼容、与发布解耦。
- [ ] 每次发布有验证过的回滚路径与自动回退触发。
- [ ] 发布前过 can-i-deploy 门禁；发布后有监控可判好坏。
- [ ] 发布可追溯（版本→commit→制品→负责人）。
