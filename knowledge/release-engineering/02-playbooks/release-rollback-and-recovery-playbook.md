# 发布回滚与恢复操作手册（商业级必读）

> 发布出事时，**最贵的是犹豫**。"再观察五分钟"往往把 5 分钟的故障拖成 1 小时。
> 这份手册是发布劣化时**照着做**的剧本：先止血、再定位、最后复盘。目标——任何人在凌晨被叫醒，照此执行也能在几分钟内让用户恢复，而不是临场发挥。

## 1. 黄金法则

- **先恢复用户，再查根因**：止血 > 调查。绝不在生产上边查边修。
- **回滚是默认动作**：发布后指标劣化，**默认回滚**，而不是默认"再看看"。回滚不需要理解根因，恢复不能等。
- **决策要预先授权**：谁有权拍板回滚、触发条件是什么，发布前就写死，事发时不用临时拉群讨论。
- **每个发布都自带回滚预案**：没有验证过的回滚路径的发布，不允许上线（见 §发布门禁）。

## 2. 止血手段优先级（从快到慢）

| 手段 | 适用 | 生效速度 | 前提 |
|---|---|---|---|
| 关功能开关（kill-switch）| 新功能藏在开关后 | 秒级 | 上线时就把功能放开关后 |
| 流量切回旧版本 | 金丝雀/蓝绿 | 秒级~分钟 | 旧版本仍在运行 |
| 回滚到上一个制品 | 滚动/常规部署 | 分钟级 | 制品不可变、可重新部署 |
| 向前修复（roll-forward）| 回滚不可行（如已改库结构）| 取决于修复 | 仅当回滚比修复更危险时 |
| 限流/降级 | 过载或部分依赖故障 | 秒级 | 预置限流/降级开关 |

原则：**能关开关就别回滚，能回滚就别向前修**——开关最快、最局部、风险最小。

## 3. 标准回滚流程（runbook）

```
0. 确认劣化   —— 看监控：错误率↑ / 延迟↑ / 核心业务指标↓，确认与本次发布相关（时间吻合）
1. 宣布并定责 —— 指定指挥(IC)，开事故频道，记录时间线（谁、何时、做了什么）
2. 立即止血   —— 按 §2 优先级选最快手段：先关开关 / 切流量 / 回滚制品
3. 验证恢复   —— 确认指标回到发布前基线；抽样验证核心用户流程真的好了
4. 稳定观察   —— 保持回滚态，禁止再次发布，直到根因清楚
5. 通告       —— 同步状态给相关方/用户（如有外部影响）
6. 复盘       —— 无指责复盘，输出根因 + 修复 + 防再发（见 incident/ 域）
```

每一步都**记时间戳**：何时发现、何时止血、何时恢复——这是事后算 MTTR 和复盘的依据。

## 4. 触发条件（自动 + 人工）

- **自动回滚/暂停**：金丝雀阶段关键指标（错误率、延迟 p95/p99、核心转化）超阈值，自动回退当前批次并告警——别靠人盯屏。
- **人工回滚**：自动没覆盖到的劣化（用户投诉激增、数据异常、下游告警）由 IC 拍板。
- **预设阈值**：发布前就定好"什么数值持续多久就回滚"，写进发布计划，事发时直接对照，不临场争论。

## 5. 数据相关发布的回滚（最危险）

代码可秒回，**数据不能**。涉及库结构/数据变更的发布要特别处理：

- **扩展-收缩护航**：变更走"先扩展（兼容新旧）→ 部署 → 稳定后再收缩"，这样代码回滚时 schema 仍兼容旧代码（见 `release-engineering/01-standards/progressive-delivery-and-release` §5）。
- **前向不兼容数据要标红**：若新版本写入了旧版本读不了的数据，**直接回滚会出错**——这类发布要么不可回滚（只能向前修），要么必须先做好双读/兼容。发布计划里提前标注"能否回滚数据"。
- **回滚前先评估数据**：执行代码回滚前，确认回滚后旧代码能否正确读取新版本期间写入的数据；不能则走向前修复。
- **变更前快照**：高风险数据变更前做备份/快照，并演练过恢复路径（光有备份没验证过恢复 = 没有备份）。

## 6. 向前修复（roll-forward）何时用

回滚**不是**永远最优。以下情况向前修复更安全：

- 已发生**不可逆**的数据写入，回滚会破坏数据一致性。
- 回滚需要的旧依赖/旧契约已不存在。
- 故障是配置/数据问题，一个小热修（hotfix）比整版回滚更快更准。

向前修复要走**最小变更 + 加速通道**：只改导致故障的那一点，走精简但必过的门禁（关键测试 + 安全扫描），不走完整发布流程，但**绝不**裸推未经测试的代码。

## 7. 演练（rollback drill）

- 回滚路径**必须在 staging/演练中验证过**才算数——没演练过的回滚，事发时大概率不灵。
- 定期做"假装发布劣化→执行回滚"的演练，校准 MTTR 和 runbook。
- 每次真实回滚后更新 runbook：哪步慢了、哪步缺工具、哪个开关没接。

## 8. 反模式（出现即不合格）

1. **舍不得回滚**：在生产上边查根因边修，把短故障拖成长事故。
2. **无验证回滚路径**：回滚预案只写在纸上，从没演练，事发时发现不灵。
3. **代码与不兼容数据强耦合发布**：一次性改 schema + 改代码，回滚即坏数据。
4. **回滚后立刻重发**：根因没清楚就再发一版，二次翻车。
5. **无 kill-switch**：新功能没放开关后，唯一止血手段是整版回滚，慢且影响面大。
6. **触发条件临场争论**：没预设阈值，事发时拉群讨论"算不算严重"，错过止血窗口。
7. **回滚不记时间线**：事后复盘无据可查，MTTR 算不出，问题反复发生。

## 9. 最低交付 checklist

- [ ] 每个发布自带**已演练**的回滚路径；未验证不允许上线。
- [ ] 新功能放在 kill-switch 后，可秒级关闭止血。
- [ ] 预设自动回滚/暂停阈值，金丝雀阶段超标自动回退。
- [ ] 回滚 runbook 明确、任何人可照做，每步记时间戳。
- [ ] 数据相关发布走扩展-收缩，前向不兼容数据已标红并评估可回滚性。
- [ ] 高风险数据变更前有备份且恢复路径演练过。
- [ ] 明确回滚决策授权人与触发条件，事发不临时讨论。
- [ ] 回滚后稳定观察、禁止盲目重发，事后做无指责复盘。
