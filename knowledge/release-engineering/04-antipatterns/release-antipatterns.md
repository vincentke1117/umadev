# 发布与回滚反模式（出现即不合格）

> 发布事故大多不是新故障，而是同一批老毛病反复犯。把它们列成"出现即不合格"的清单，评审/门禁直接拦。
> 配套规范见 `release-engineering/01-standards/progressive-delivery-and-release` 与回滚手册 `02-playbooks/release-rollback-and-recovery-playbook`。

## 1. 发布动作类

| 反模式 | 为什么危险 | 正解 |
|---|---|---|
| **全量直发** | 100% 流量一把切，无灰度无观测，出事即全员受影响 | 金丝雀/蓝绿分步放量，按指标晋级 |
| **大爆炸发布** | 攒一个月一次性上，变更面巨大，出事无法定位 | 小步快跑、高频小发布 |
| **周五/深夜大发布** | 出事时人最少、响应最慢 | 规避高风险时段，留足值守 |
| **环境间重新构建** | staging 和 prod 跑的不是同一个制品 | 一次构建、不可变制品、跨环境晋级同一份 |
| **发布无观测** | 上完线靠用户投诉才知道坏了 | 发布前接好错误率/延迟/业务指标 |
| **手工发布步骤** | 靠人记命令，易漏易错、不可复现 | 发布自动化、声明式、可重复 |

## 2. 回滚类

| 反模式 | 为什么危险 | 正解 |
|---|---|---|
| **无回滚预案** | 出事现想办法，止血慢 | 每个发布自带已验证回滚路径 |
| **回滚没演练** | 纸面计划事发不灵 | staging/演练实际跑过回滚 |
| **舍不得回滚** | 在生产边查边修，短故障拖成长事故 | 默认回滚，先恢复用户再查根因 |
| **回滚后立刻重发** | 根因没清就再发，二次翻车 | 稳定观察、禁止盲目重发 |
| **触发条件临场争论** | 没预设阈值，错过止血窗口 | 发布前写死触发阈值与授权人 |
| **回滚不记时间线** | 复盘无据，问题反复 | 每步记时间戳，事后复盘 |

## 3. 数据与耦合类

| 反模式 | 为什么危险 | 正解 |
|---|---|---|
| **部署即发布、强耦合** | 代码与不兼容 schema 一次性上，回滚即坏数据 | 解耦部署与发布，schema 走扩展-收缩 |
| **不可逆变更混进普通发布** | 删字段/数据迁移和常规发版一起上，回不去 | 不可逆变更单独走、提前演练、标红 |
| **前向不兼容数据不评估** | 新版写入旧版读不了的数据，回滚出错 | 发布前评估可回滚性，必要时双读兼容 |
| **大回填阻塞线上** | 迁移锁表/打满 IO 拖垮生产 | 回填与发布分离，可中断、可重入、限速 |
| **备份没验证恢复** | 有备份但恢复路径从没跑通 = 没有备份 | 恢复路径定期演练 |

## 4. 功能开关类

| 反模式 | 为什么危险 | 正解 |
|---|---|---|
| **开关债** | 临时开关从不清理，代码堆满死分支 | 上线稳定后及时清理临时开关 |
| **无 kill-switch** | 新功能没放开关后，唯一止血是整版回滚 | 风险功能上线即放开关后 |
| **开关默认不安全** | 默认开 / 默认放量，新代码一上即全量暴露 | 默认关 / 默认灰度 |
| **关键开关无审计** | 谁改了开关、何时改，无记录 | 关键开关变更留审计 |

## 5. 版本与契约类

| 反模式 | 为什么危险 | 正解 |
|---|---|---|
| **破坏性变更不升版** | 直接改线上契约，消费方大面积中断 | 破坏性变更升主版本 + 过渡期 |
| **changelog 堆 commit** | 使用者读不懂，升级风险不明 | 面向使用者写新增/变更/修复/破坏性/弃用 |
| **制品不可追溯** | 出事查不到对应 commit/构建 | 制品可追溯到 commit、构建、依赖清单 |
