# 发布回滚就绪 checklist（上线前逐条过）

> 这是发布"放行"前必须逐条勾过的清单，配套手册见 `release-engineering/02-playbooks/release-rollback-and-recovery-playbook`。
> 任何一条没勾，发布**不放行**。回滚能力不是事后补的，是上线的入场券。

## 1. 回滚路径

- [ ] 本次发布的回滚方式已明确（关开关 / 切流量 / 回滚制品 / 向前修复中的哪一种）。
- [ ] 回滚路径已在 staging 或演练中**实际执行验证过**，不是纸面计划。
- [ ] 回滚预计耗时已知（秒级 / 分钟级），并满足止血时效要求。
- [ ] 旧版本制品仍可用、可重新部署（制品不可变、有版本号）。

## 2. 开关与流量

- [ ] 新功能藏在功能开关（kill-switch）后，可独立于发版秒级关闭。
- [ ] 开关默认值安全（默认关 / 灰度），关闭路径已测试。
- [ ] 采用金丝雀/蓝绿，旧版本在回滚窗口内保持可承接流量。

## 3. 自动触发

- [ ] 已设自动回滚/暂停阈值（错误率、p95/p99 延迟、核心业务指标）。
- [ ] 金丝雀各放量步骤有晋级判据与自动回退条件，不靠人盯。
- [ ] 告警已接好，发布后能立刻从监控看出好坏。

## 4. 数据安全

- [ ] 库结构/数据变更走扩展-收缩，向后兼容，回滚后旧代码仍能读。
- [ ] 是否存在"前向不兼容数据"（新版写入旧版读不了）已评估并标注。
- [ ] 不可回滚的数据变更已单独标红，并准备了向前修复方案。
- [ ] 高风险数据变更前已备份/快照，且恢复路径演练过。

## 5. 人员与流程

- [ ] 回滚决策授权人明确（谁有权拍板）。
- [ ] 触发回滚的条件/阈值已预先写定，事发无需临时讨论。
- [ ] 发布负责人 + 回退负责人 + on-call 明确，联系方式可达。
- [ ] runbook 任何 on-call 可照做，无需作者在场。

## 6. 门禁（与 can-i-deploy 一致）

- [ ] 所有必过 CI 检查通过（测试 / 契约 / 构建 / 安全扫描）。
- [ ] 契约兼容当前线上所有消费方，无未升版的破坏性变更。
- [ ] 跨环境晋级的是**同一个**不可变制品（未重新构建）。
- [ ] 高风险时段 / 变更冻结窗口已规避。
- [ ] 发布有记录（谁、何时、什么版本、对应 commit）。

## 7. 发布后（回滚演练复盘前置）

- [ ] 发布后保留观察窗口，期间禁止叠加新发布。
- [ ] 若发生回滚，记录完整时间线（发现→止血→恢复）。
- [ ] 回滚后根因未清前不重发；复盘输出修复 + 防再发项。
