# 授权与访问控制模型规范（商业级必读）

> 认证（你是谁）只是门票，**授权（你能做什么、能动哪条数据）才是商业级安全的真正战场**。越权（Broken Access Control）长期是 Web 应用头号风险。本规范是框架无关的硬性约束：每个写操作要鉴权限、鉴资源归属、鉴租户；授权决策要集中、可审计、默认拒绝。认证实现见 `development/01-standards/authentication-patterns-complete.md`，本篇只讲授权。

## 1. 三个绝不能省的检查（每个敏感请求都要过）

每个访问受保护资源的请求，服务端必须独立验证三件事——**前端隐藏按钮不算授权**：

1. **能力检查（Permission）**：当前主体是否有执行该动作的权限（`order:cancel`）。
2. **资源归属检查（Ownership / Tenancy）**：该资源是否属于当前主体/租户（防 IDOR——改 URL 里的 id 就看到别人的单）。
3. **状态/条件检查（Context）**：当前业务状态是否允许（已发货的单不能改地址；超额度不能审批）。
- **默认拒绝（deny by default）**：没有明确 allow 就是 deny。新增端点默认无人可访问，再按需放开。
- 授权在**服务端 + 数据访问层**双重落地，绝不信任客户端传来的角色/租户字段。

## 2. 主流模型与选型（RBAC / ABAC / ReBAC）

- **RBAC（基于角色）**：用户→角色→权限。简单、好理解、好审计，适合**角色清晰、扁平、单租户**的后台。缺点：层级、共享、跨租户场景会"角色爆炸"。
- **ABAC（基于属性）**：用主体属性（部门、等级）+ 资源属性（密级、归属）+ 环境（时间、IP）+ 动作，写成策略规则求值。适合**动态、细粒度、条件式**的访问（"同部门且工作时间可读"）。缺点：策略多了难审计"谁能看这条"。
- **ReBAC（基于关系）**：用实体间关系判定权限（"X 是该文档所在文件夹的 editor"→可编辑）。天然支持**层级继承、资源共享、嵌套组织**，是协作类/多层级资源产品的最佳解；它是 RBAC 的超集，属性表达为关系时也能覆盖 ABAC 场景。缺点：需要一套关系存储与图查询。
- **选型基线**：
  - 角色固定的内部后台 → **RBAC** 足够。
  - 规则随属性/条件变化 → **RBAC 打底（粗粒度） + ABAC 细化**。
  - 文档/项目/组织树这类"谁能访问取决于关系" → **ReBAC**。
  - 别一上来就追最复杂的；**从 RBAC 起步，按真实需求向 ABAC/ReBAC 演进**，三者可组合。

## 3. 权限模型设计要点

- 权限粒度用 **`资源:动作`**（`invoice:read` / `invoice:approve`），不要用"页面"做权限单位。
- 角色是**权限的集合**，给用户配角色，不直接配权限；角色要贴业务（`finance.approver`），别叫 `role1`。
- 区分**全局角色**与**资源域角色**（在 A 项目是 admin、在 B 项目是 viewer）——多数 SaaS 需要后者，提前设计。
- 维护**权限矩阵**（角色 × 资源 × 动作）作为评审与测试的依据；它也是测试用例的来源。
- 危险动作（删除、导出、提权、改权限、转账）要**额外门槛**：二次确认 / 审批流 / 重新认证（step-up auth）。
- **最小权限 + 职责分离**：默认给最小集；申请提权要审批、要到期回收；申请者≠审批者。

## 4. 集中式授权：策略与执行点分离

商业级系统不要把授权逻辑散在每个 `if` 里——**集中决策、分散执行**：

- **决策点（PDP）**与**执行点（PEP）**分离：业务代码只问"这个主体能否对这个资源做这个动作？"，由统一的授权组件/策略引擎回答。
- 策略**外置、可版本化、可测试**（策略即代码）：策略变更走 PR + 测试，而非改散落代码。
- 决策要**可解释**：能回答"为什么 allow/deny"，便于排障与审计。
- 性能：授权在每个请求热路径上，**缓存关系/角色解析结果**（带失效），批量端点要批量鉴权，避免 N 次远程策略调用。
- **fail-closed**：授权服务/策略求值异常时**拒绝**而非放行（与治理类 fail-open 相反——授权必须保守）。

## 5. 多租户隔离（SaaS 必做）

- **每个查询都带租户维度**：`WHERE tenant_id = ? AND id = ?`，绝不只按主键查。强烈建议在数据访问层/ORM 统一注入租户过滤，靠人工每次加 where 必出事。
- 隔离强度按敏感度选：共享库共享表（行级 + 强制租户过滤）→ 共享库独立 schema → 独立库。金融/医疗类倾向更强隔离。
- 数据库层加固：行级安全（RLS）作为最后一道防线，即使应用漏加 where 也兜底。
- **跨租户访问是默认禁止的**；平台超级管理员的跨租户操作要单独审计、单独提权、单独留痕。
- 缓存键、文件路径、消息队列、搜索索引**都要带租户隔离**——别只防住了数据库却在缓存/对象存储里串号。

## 6. 令牌、会话与权限传播

- 访问令牌里只放**稳定的身份与粗粒度授权**（user_id、tenant_id、roles/scopes），细粒度资源权限运行时查，避免令牌过期信息滞后。
- 权限/角色变更要能**快速生效**：缩短访问令牌有效期 + 刷新令牌可吊销 + 必要时维护吊销列表；高危场景用即时校验。
- `scope`（OAuth）控制"第三方应用代表用户能做什么"，与用户自身权限是**两层**，最终权限取两者交集。
- 服务间调用要传播并**重新校验**调用方身份与权限，不要因为"内网调用"就默认可信（零信任，见 `security/01-standards/zero-trust-architecture.md`）。

## 7. 反模式（出现即不合格）

- **只在前端控制**：靠隐藏按钮/路由守卫，后端不校验——改请求即越权。
- **缺资源归属校验（IDOR）**：`GET /api/orders/{id}` 只查主键不查归属，遍历 id 就能拖库。
- **默认放行**：新端点忘了加鉴权就对所有人开放（应默认拒绝）。
- **角色硬编码散落**：`if user.role == "admin"` 满天飞，改权限要全局搜代码。
- **多租户漏租户过滤**：查询只按主键，靠开发记得加 `tenant_id`。
- **越权信息泄露**：用"403 vs 404"或错误文案暴露"该资源存在但你没权限"，给攻击者枚举线索（按策略统一处理）。
- **授权 fail-open**：策略服务超时就放行。
- **提权不回收**：临时提权/借权没到期机制，权限只增不减。
- **批量/导出/GraphQL 绕过**：单条鉴权了，批量接口或嵌套查询没逐条鉴权。

## 8. Agent Checklist（接口/数据访问前必过）

- [ ] 每个敏感请求是否在服务端做了能力 + 资源归属 + 状态三重校验？
- [ ] 是否默认拒绝（新端点不显式放开即不可访问）？
- [ ] 授权模型是否与业务匹配（RBAC 起步，按需 ABAC/ReBAC），而非过度或不足？
- [ ] 权限是否为 `资源:动作` 粒度，有可评审的权限矩阵？
- [ ] 授权决策是否集中（PDP/PEP 分离）、策略可测试、可解释、fail-closed？
- [ ] 多租户：每个查询/缓存/文件/索引是否都带租户隔离，DB 是否有 RLS 兜底？
- [ ] 高危动作是否有二次确认/审批/step-up？职责分离是否落实？
- [ ] 权限变更能否快速生效（短令牌 + 可吊销）？
- [ ] 批量/导出/嵌套查询是否逐条鉴权，未被绕过？
- [ ] 越权与不存在是否统一响应，未泄露资源存在性？
