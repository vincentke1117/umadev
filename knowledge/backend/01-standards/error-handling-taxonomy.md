# 错误处理分类与用户可见错误契约（商业级必读）

> "出错就 try/catch 打条日志返回 500"是业余做法。商业级系统先把错误**分类**——这是 bug 还是预期内的故障？能不能重试？该不该让用户看到？——再据此决定**快速失败、重试、降级，还是向用户解释**。
> HTTP 错误信封（见 `backend/01-standards/api-and-error-conventions`）解决"错误长什么样"；本规范解决更上游的问题：**如何对错误分类，以及给最终用户看什么**。

## 1. 第一刀：可预期 vs 不可预期

| 类别 | 含义 | 处理方式 |
|---|---|---|
| **操作性错误（operational）**| 正确程序在运行时遇到的**预期内**异常：网络超时、依赖不可用、输入非法、余额不足、限流 | 预先设计好如何处理：校验、重试、降级、向用户解释 |
| **程序性错误（programmer / bug）**| 代码缺陷：空指针、类型错、断言失败、不该到达的分支 | **不要吞**。让它快速失败、记录、告警、修复——重试或掩盖只会掩埋 bug |

这一刀是基础：操作性错误要**优雅处理**，程序性错误要**快速暴露**。把 bug 当操作性错误吞掉（"catch 一切返回默认值"）是最常见的腐化源——故障被静默，数据被悄悄写坏。

## 2. 第二刀：按来源与责任分类

| 类型 | 例子 | 状态码倾向 | 给用户看 |
|---|---|---|---|
| 用户错误（user error）| 表单非法、缺字段、格式错 | 4xx（422/400）| **要**：明确告诉哪里错、怎么改 |
| 鉴权/授权错误 | 未登录、无权限 | 401 / 403 | 要：提示登录或无权，但不泄露资源是否存在 |
| 业务规则拒绝 | 库存不足、状态不允许、超额 | 409 / 422 | 要：说明业务原因（可执行）|
| 资源不存在 | 找不到 | 404 | 要：简洁提示 |
| 限流 | 触发速率限制 | 429 | 要：提示稍后重试（可给 Retry-After）|
| 依赖故障（下游）| 第三方/数据库挂 | 503 / 502 | **泛化**：暂不可用，请稍后；不暴露内部 |
| 内部缺陷（bug）| 未捕获异常 | 500 | **泛化**：通用错误 + request_id；绝不暴露栈/SQL |

原则：**4xx 是调用方的问题（可指明并指导），5xx 是我们的问题（对用户泛化，对自己详细记录 + 告警）。**

## 3. 第三刀：可重试 vs 终止性

- **可重试（transient）**：超时、连接失败、5xx/服务不可用、限流——配合指数退避 + 抖动 + 重试预算可重试（见 `backend/01-standards/resilience-and-fault-tolerance`）。
- **终止性（permanent）**：4xx 参数错、鉴权失败、业务拒绝——**重试也白搭**，还放大压力；直接返回，让调用方修正。
- 在错误类型/错误码上**显式标注**可否重试，让调用方和重试中间件能据此决策，而不是猜。

## 4. 用户可见错误契约（user-facing error contract）

给最终用户的错误信息是产品体验的一部分，不是技术副产物。一条合格的用户错误信息：

| 要素 | 要求 |
|---|---|
| 说人话 | 不暴露栈、SQL、内部码、英文异常名；用用户能懂的语言 |
| 可执行 | 告诉用户**下一步怎么办**（改什么、稍后重试、联系支持）|
| 不吓人、不甩锅 | 系统侧故障别让用户觉得是自己错了 |
| 不泄密 | 不暴露内部结构、是否存在某资源、为何鉴权失败的细节 |
| 可定位 | 携带 `request_id`/支持码，用户报障时工程师能据此查到那条日志/审计 |
| 可国际化 | 文案走 i18n，按 `code` 渲染，不在代码里硬编码某语言句子 |

机器可读 + 人类可读分离：接口返回**稳定的机器码**（`INSUFFICIENT_STOCK`）+ 文案；前端按 `code` 分支处理与本地化，**不靠 message 文案做逻辑**（文案会变、会翻译）。

## 5. 分层处理职责

- **边界校验**：在入口（API 层）校验输入，非法即 422 + 字段错误，别让脏数据流进核心。
- **集中映射**：一个统一的错误处理器/过滤器把内部异常映射成对外错误信封，**不在每个 controller 重复 try/catch**。
- **保留上下文**：异常向上传递时保留原因链（cause），别 catch 后只 `throw new Error("failed")` 丢掉根因。
- **区分日志级别**：用户错误/业务拒绝 → INFO/WARN（这是正常流程，不是事故）；bug/依赖故障 → ERROR + 告警。别把每个 422 都打成 ERROR 淹没真问题。
- **失败要可观测**：错误带 `request_id`、错误码、关键上下文进结构化日志，便于聚合定位（见 `observability/`）。

## 6. 反模式（出现即不合格）

1. **吞异常**：`catch` 后什么都不做 / 返回默认值，把 bug 和故障静默掉。
2. **一律 500 或一律 200**：所有错误都 500，或业务失败回 200 再在 body 塞 `success:false`。
3. **向用户泄露内部**：把栈、SQL、内部路径、异常类名直接抛到前端。
4. **吓人/无指引文案**："Error" / "Something went wrong" 没有下一步。
5. **靠 message 做逻辑**：前端用文案字符串匹配判断错误类型（一改文案/一翻译就崩）。
6. **该 fail-fast 却重试**：对 bug 或终止性错误反复重试，掩埋问题、放大压力。
7. **丢根因**：层层 catch-rethrow 把原始原因链丢光，线上无法定位。
8. **422 当 ERROR 刷屏**：正常的用户输入错误被打成 ERROR，把日志淹没、把告警拖疲。

## 7. 最低交付 checklist

- [ ] 区分操作性错误（优雅处理）与程序性 bug（快速失败 + 告警，绝不吞）。
- [ ] 按来源分类映射到正确状态码：4xx 调用方问题、5xx 我方问题。
- [ ] 错误显式标注可重试/终止性，配合重试预算使用。
- [ ] 对用户：泛化内部故障、不泄露内部、给可执行下一步、带 request_id。
- [ ] 返回稳定机器码 + i18n 文案，前端按 code 分支，不靠 message。
- [ ] 输入边界校验 + 集中错误映射，不在各处重复 try/catch。
- [ ] 异常保留原因链；按类型分日志级别，bug/依赖故障告警。
- [ ] 错误进结构化日志，可聚合、可按 request_id 定位。
