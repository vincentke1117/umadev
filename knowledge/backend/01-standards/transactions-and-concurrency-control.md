# 事务与并发控制规范（商业级后端必读）

> 多个请求同时改同一份数据时，"看起来能跑"的代码会丢更新、读脏数、超卖。
> 商业级后端必须显式定义**事务边界**、选对**隔离级别**、用对**锁策略**。
> 本规范聚焦单库内的本地事务与并发；跨服务/跨库一致性见 `architecture/distributed-transactions`。

## 1. 事务边界 = 用例边界

- **一个用例（业务操作）= 一个事务**。在**应用层/服务层**开启与提交事务，而不是在 repository 里各开各的（否则一个用例里多次提交，失败时一半已落库）。
- 事务要**短小**：只把"必须原子"的读写包进去。**绝不**在事务里发 HTTP 调用、发消息、等外部 IO——会长时间持锁、放大死锁与连接耗尽。需要对外副作用时，用发件箱（outbox）在事务内只写消息，事务外再投递。
- 事务里**先做校验/计算，再做写**，把写集中在末尾，缩短持锁时间。
- 失败必须**整体回滚**；不要 catch 异常后继续提交"半截"数据。

## 2. ACID 与一致性底线

- **原子性**：要么全成功要么全回滚，没有"半完成"。
- **一致性**：事务结束后所有约束（唯一/外键/check/余额非负）都成立——把不变量尽量交给 **DB 约束**，不要只在应用层校验。
- **隔离性**：并发事务互不干扰的程度，由隔离级别决定（见 §3）。
- **持久性**：提交即落盘，宕机不丢。
- 金额用**定点/整数（分）**，绝不用浮点；时间带时区；并发计数器避免"读-改-写"丢更新。

## 3. 隔离级别与并发异常

选错隔离级别会出现这些异常，按场景选最低够用的级别：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 写偏斜(write skew) | 典型用途 |
|---|---|---|---|---|---|
| Read Uncommitted | 可能 | 可能 | 可能 | 可能 | 几乎不用 |
| Read Committed | 否 | 可能 | 可能 | 可能 | OLTP 常见默认 |
| Repeatable Read | 否 | 否 | 可能/否* | 可能 | 需要稳定快照的读 |
| Serializable | 否 | 否 | 否 | 否 | 强一致关键写（账务/库存判定） |

\* 不同数据库在 Repeatable Read 下对幻读的处理不同；需要严格不变量时用 Serializable 或显式加锁。

- 异常含义：**脏读**=读到未提交数据；**不可重复读**=同一行两次读值不同；**幻读**=同一查询条件两次返回行数不同；**写偏斜**=两个事务各自读一致快照后分别写，合起来违反不变量（如"至少留一个值班医生"被两人同时请假打破）。
- 经验：默认 **Read Committed**；对"读后据此决定写"的关键判定（余额、库存、配额），升到 **Serializable** 或用显式锁，否则会有写偏斜/超卖。

## 4. 乐观锁 vs 悲观锁

| 维度 | 乐观锁 | 悲观锁 |
|---|---|---|
| 机制 | 版本号/时间戳，提交时 CAS 校验 | `SELECT ... FOR UPDATE` 先锁后改 |
| 适用 | 冲突少、读多写少 | 冲突多、临界区短、必须串行 |
| 失败处理 | 影响行数=0 → 重读重试（带上限） | 等待锁/超时 |
| 风险 | 高冲突下大量重试 | 持锁久 → 死锁、吞吐下降 |

- **乐观锁**：`UPDATE t SET val=?, version=version+1 WHERE id=? AND version=?`；影响行数为 0 表示已被他人修改，重读后按业务决定重试或放弃。**先查后写**前必须有这层校验，否则丢更新。
- **悲观锁**：`SELECT ... FOR UPDATE` 锁定将要修改的行，确保串行。务必：① 加锁范围最小；② 设锁等待超时；③ **所有事务以相同顺序加锁**以避免死锁。
- **原子写优先**：能用一条原子语句就别先查后写——如 `UPDATE stock SET qty=qty-1 WHERE id=? AND qty>=1`（影响行数=0 即库存不足），既无竞态又免去显式锁。

## 5. 死锁与锁竞争

- **死锁成因**：两个事务以**相反顺序**持有/请求锁。根治：固定全局加锁顺序（如总是按主键升序锁多行）。
- 数据库会检测死锁并回滚其中一个事务——业务侧要**捕获死锁错误并重试**（幂等前提下，带退避）。
- 减少锁竞争：缩短事务、降低隔离级别到够用、给热点行设计分桶/分片、把"读多"路径走快照读不加锁。
- **连接池**：事务持有连接，长事务会拖垮连接池；设置语句/事务超时，杜绝"忘了提交"的悬挂事务。

## 6. 读一致性

- **读己之写（read-your-writes）**：用户写完立刻读应看到自己的写；主从架构下要把"刚写完的读"路由到主库或加版本校验，避免读到滞后的从库。
- 报表/聚合类长读用**快照读**或专门的只读副本，别和 OLTP 写抢锁。
- 需要"一批读相互一致"时，把它们放进同一个（Repeatable Read/Serializable）事务的同一快照里。

## 7. 反模式（出现即不合格）

1. **事务里做外部 IO**：在事务内发 HTTP/MQ，长持锁、易死锁。
2. **repository 各自提交**：一个用例多个事务，失败留下半截数据。
3. **先查后写无并发保护**：丢更新、超卖（缺乏唯一约束/乐观锁/原子写）。
4. **隔离级别不假思索**：关键判定用 Read Committed 出现写偏斜。
5. **金额用浮点 / 计数器读改写**：精度丢失、并发丢更新。
6. **长事务 + 大锁范围**：吞吐塌方、死锁频发、连接耗尽。
7. **不处理死锁/锁超时**：偶发死锁直接变成用户报错。

## 8. 最低交付 checklist

- [ ] 事务边界在服务层、对齐用例；事务短小、无内部外部 IO。
- [ ] 隔离级别按场景显式选择；关键"读后写"判定有写偏斜防护。
- [ ] 并发写有保护：原子写 / 乐观锁（版本号）/ 悲观锁，三选一且最小化。
- [ ] 多行加锁顺序固定；捕获死锁与锁超时并安全重试。
- [ ] 不变量尽量落 DB 约束（唯一/外键/check），不只应用层校验。
- [ ] 金额用整数/定点；计数器用原子更新而非读-改-写。
- [ ] 有并发场景的自动化测试（并发下单/扣减/转账不出错）。
