# 幂等性与安全重试规范（商业级后端必读）

> 网络会超时、客户端会重试、消息队列会重投——任何一次写操作都可能被执行**不止一次**。
> 商业级后端必须保证"重复执行不产生重复副作用"：一次下单不能扣两次款，一条消息不能发两遍。
> 不做幂等的写接口，在生产里一定会出资损或脏数据。

## 1. 核心定义

- **幂等（idempotent）**：同一请求执行一次和执行多次，对系统状态的影响**相同**。读、`PUT` 全量替换、`DELETE` 天然幂等；`POST` 创建/扣款**默认不幂等**，必须人为做成幂等。
- **安全重试**：客户端/调用方在不确定上一次是否成功时，可以**放心重试**而不会造成第二次副作用。
- **投递语义**：分布式下"恰好一次（exactly-once）投递"几乎不可达；现实是"至少一次（at-least-once）投递 + 消费端幂等 = 有效恰好一次（effectively-once）处理"。

## 2. 哪些操作必须做幂等

| 操作 | 是否需要幂等 | 原因 |
|---|---|---|
| 创建订单 / 支付 / 扣款 / 退款 | 必须 | 重复=资损 |
| 发短信/邮件/推送/通知 | 必须 | 重复=骚扰用户 |
| 扣减库存 / 发放优惠券/积分 | 必须 | 重复=超卖/多发 |
| 消费 MQ 消息（任意业务） | 必须 | 队列至少一次投递 |
| Webhook 回调处理 | 必须 | 上游会重投 |
| 纯查询 / 全量替换 PUT | 天然幂等 | 无累积副作用 |

## 3. 幂等键（Idempotency Key）——首选方案

让**调用方生成**一个唯一键（UUID/业务单号），随请求传入（`Idempotency-Key` 头或请求体字段）。服务端用它去重：

```
1. 收到请求，取出幂等键 K。
2. 在去重存储里以 K 为唯一约束尝试占位（INSERT ... ON CONFLICT / SETNX）。
3. 占位成功（首次）→ 执行业务，把"结果"与 K 一起持久化，返回结果。
4. 占位失败（重复）→ 不再执行业务，直接返回上次已保存的结果（同样的状态码与响应体）。
```

要点：
- 幂等键要带**作用域**：通常 `(用户/租户, 业务类型, K)` 组合唯一，避免跨业务撞键。
- 去重记录要**存结果**，重复请求要返回**与首次一致**的响应（含相同的 `201/200` 与 body），而不是报错。
- 给去重记录设 **TTL**（如 24h~7d），匹配业务"可能重试的窗口"，避免无限增长。
- "占位"和"业务写"要么在**同一事务**里，要么用"先占位(pending)→执行→标记(done)"的状态机，并处理"占位了但业务还没跑完又来一次"的并发（见 §5）。

## 4. 没有幂等键时的去重手段

调用方无法传键时，用**业务自然唯一性**兜底：

- **唯一约束**：在 DB 上对业务唯一维度建唯一索引（如 `(order_no)`、`(user_id, idempotency_scope, day)`），重复插入直接被 DB 拒绝（捕获唯一冲突当作"已处理"）。这是最可靠的最后一道防线——**不要只靠"先查再插"**（查与插之间有竞态）。
- **去重表 / 去重缓存**：以"消息ID/事件ID/请求指纹"为键记录已处理，消费前先查。缓存去重要接受偶发漏判，**关键资损场景必须落 DB 唯一约束**。
- **状态机前置校验**：只允许从合法前态流转（`pending → paid`），重复的"支付成功"在 `paid` 态下变成无副作用的空操作。

## 5. 并发与"进行中"的重复

重复请求可能**几乎同时**到达，"先查后写"会双双查到"没处理过"然后都执行。必须用以下之一根除竞态：

- **DB 唯一约束**：让数据库串行化，第二个插入失败。
- **占位即加锁**：插入一条 `status=processing` 的去重记录（唯一键），第二个请求插入失败→说明"有人正在处理"，返回 409/稍后重试或等待首个结果。
- **乐观锁（版本号/CAS）**：`UPDATE ... SET version=version+1 WHERE id=? AND version=?`，影响行数为 0 说明已被改过，放弃或重读重试。
- **悲观锁 / 分布式锁**：`SELECT ... FOR UPDATE` 或带 TTL 的分布式锁，锁的粒度要小、要有超时、要可重入安全。分布式锁必须配合**栅栏令牌（fencing token）**或 DB 唯一约束兜底，因为锁会因 GC/超时被两个持有者同时认为自己持有。

## 6. 安全重试的客户端侧

- 只对**幂等操作**自动重试；非幂等操作重试前必须先变幂等（带幂等键）。
- 重试用**指数退避 + 抖动（jitter）**，设最大次数与总时限，避免重试风暴打垮下游（参见 `architecture/resilience-and-disaster-patterns`）。
- 区分错误：`408/429/5xx/网络超时`可重试；`4xx`（除 429）是请求本身的问题，**不要重试**。
- 超时不等于失败：超时后状态**未知**，必须靠幂等键安全重试来确认，而不是盲目再发一笔。

## 7. 消息与跨服务：Outbox + 消费端幂等

异步链路要保证"本地状态变更"与"对外消息"原子一致，且消费端不重复处理：

- **事务发件箱（Outbox）**：在同一本地事务里写业务表 + 写一条 outbox 消息；独立投递器再把 outbox 发到 MQ。避免"DB 提交了但消息没发"或"消息发了但 DB 回滚了"的双写不一致。
- **消费端去重（Inbox）**：消费者以"消息ID/事件ID"为唯一键记录已处理，重复投递被去重表挡掉。
- 结论：**生产端至少一次 + 消费端幂等 = 有效恰好一次**。不要追求传输层的"恰好一次"。

## 8. 反模式（出现即不合格）

1. **写接口不幂等**：`POST /pay` 重复调用扣两次款。
2. **只靠"先查再插"去重**：查与插之间有竞态，并发下双写。
3. **幂等键无 TTL / 无作用域**：存储无限增长，或跨业务撞键。
4. **重复请求报错而非回放结果**：客户端重试拿到 409 后无法判断到底成没成。
5. **对非幂等操作自动重试**：超时即重发，制造重复副作用。
6. **重试无退避无上限**：下游抖动时把它彻底打死（重试风暴）。
7. **依赖分布式锁但无 DB 唯一兜底**：锁失效时两个持有者同时写。

## 9. 最低交付 checklist

- [ ] 所有"创建/扣款/发券/发通知/消费消息"类写操作都做了幂等。
- [ ] 资损相关路径有 **DB 唯一约束**作为最终防线，不只靠缓存/先查后插。
- [ ] 幂等键带作用域 + TTL，重复请求回放首次结果（状态码与 body 一致）。
- [ ] 并发重复用唯一约束/乐观锁/占位锁根除竞态，覆盖"进行中"重复。
- [ ] 客户端重试仅限幂等操作，带指数退避+抖动+上限，区分可重试错误。
- [ ] 跨服务用 Outbox 保双写一致 + 消费端 Inbox 去重。
- [ ] 有针对"重复提交/超时重试/消息重投"的自动化测试。
