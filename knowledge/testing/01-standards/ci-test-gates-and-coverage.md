# CI 测试门禁与覆盖率规范（商业级必读）

> 写了测试但不在 CI 里强制跑、不阻断合并，等于没写。商业级交付要求：**每次改动都自动验证，红就不能合**。
> 本规范定义哪些检查在哪个阶段跑、覆盖率怎么卡、flaky 怎么治，让"绿色"真正可信。

## 1. 门禁原则

- **快速反馈分层**：把检查按"快→慢"分层，快的先挂、早失败（fail fast），别让开发者等 30 分钟才看到一个拼写错。
- **必过即阻断**：被列为"required check"的项失败，PR **不可合并**——靠分支保护强制，而不是靠自觉。
- **可信的绿**：绿色必须代表"真的没问题"。flaky（随机红/绿）会侵蚀信任，让人习惯性"重跑一下"，必须当一类缺陷治理。
- **可复现**：CI 跑的就是别人本地能跑的同一套命令；锁依赖版本、固定随机种子与时钟。

## 2. 分阶段执行什么

| 阶段 | 触发 | 跑什么 | 目标时长 |
|---|---|---|---|
| 提交前 / 本地钩子 | commit/push | 格式化、lint、改动文件的快速单测 | 秒级 |
| PR（必过门禁）| 每次 PR & 更新 | lint + 类型检查 + 全量单测 + 集成测 + 契约校验 + 构建 + SAST/依赖扫描 | < 10 分钟 |
| 合并到主干 | merge | 上面全部 + 关键路径 E2E 冒烟 | < 20 分钟 |
| 夜间 / 定时 | 定时 | 全量 E2E、性能基准、变异测试、可视回归、SCA 全扫 | 不阻断开发 |

要点：**慢而全的**（完整 E2E、性能、变异测试）放夜间，避免拖慢每次 PR；**快而关键的**（单测、集成、契约、lint、构建）进 PR 必过门禁。

## 3. PR 必过检查清单（required checks）

每个 PR 合并前必须全绿：
- [ ] 代码格式化无 diff（formatter 干净）。
- [ ] Lint 0 错误（pedantic/严格级别，警告即错误）。
- [ ] 静态类型检查通过。
- [ ] 全量单元测试通过。
- [ ] 集成测试通过（含真实 DB/依赖的关键读写）。
- [ ] 接口契约校验通过（见 `contract-testing-and-api-contracts`）。
- [ ] 项目构建成功（产物可生成）。
- [ ] 依赖漏洞扫描 + 静态安全扫描（SAST）无新增高危。
- [ ] 覆盖率达标（见 §4），且**新增/改动代码**覆盖率达阈值。
- [ ] 无密钥/凭据泄漏（secret 扫描）。

## 4. 覆盖率怎么卡

覆盖率是**护栏不是目标**——100% 行覆盖也能全是无断言的假测试。正确用法：

- **关注变更覆盖（diff/patch coverage）**：本次改动的新增/修改代码覆盖率必须达阈值（如 ≥ 80%）。这比"全库总覆盖率"更能防止"新代码不写测试"。
- **设全局下限 + 禁止下降**：总覆盖率有底线（如关键模块 ≥ 80%、整体 ≥ 70%），且 PR 不得让覆盖率**净下降**。
- **分层看覆盖**：核心领域逻辑/计费/权限要求高覆盖（含分支）；UI 胶水、生成代码、配置可豁免。
- **分支覆盖 > 行覆盖**：行覆盖会漏掉未走到的条件分支；对关键逻辑要求分支覆盖。
- **变异测试做抽查（夜间）**：行覆盖只证明"代码被执行"，变异测试改变代码看测试是否能发现，证明"断言真的有效"。对核心模块定期跑，揪出"覆盖了但没断言"的空测试。

覆盖率反模式：为冲数字写无断言测试、把难测代码排除统计、只看总数不看变更覆盖。

## 5. Flaky 测试治理

flaky 是 CI 的头号信任杀手，按缺陷对待：
- **检测**：同一提交多次运行结果不一致即 flaky；记录失败率。
- **隔离（quarantine）**：确认 flaky 后立刻移出必过门禁（打标隔离）、建工单限期修，**但不允许**长期挂着无人管。
- **根因**：常见是时间/随机/并发顺序依赖、共享可变状态、真网络、未等待异步、测试间数据污染。修复方向：注入可控时钟与种子、每用例独立数据与事务回滚、显式等待替代 sleep、并行安全。
- **禁止"重跑掩盖"**：把"重试直到绿"当默认策略会永久隐藏真 bug；自动重试只用于明确隔离区，且计数告警。

## 6. 测试数据与环境

- 集成测试用**一次性真实依赖**（容器化 DB/内存库），每用例自带数据、跑完清理或事务回滚，互不污染。
- 不依赖共享/生产测试库；不依赖"先跑某测试"的隐式顺序。
- 测试可**并行**：无全局可变状态、无固定端口冲突、数据隔离。
- 确定性：固定随机种子、可注入时钟、稳定排序，杜绝时序/时区导致的 flaky。

## 7. 反模式（出现即不合格）

1. **写了测试不进 CI**：本地偶尔跑，主干随时可坏。
2. **检查非必过**：失败也能合并（没接分支保护）。
3. **覆盖率只看总数**：新代码零测试但总数没掉，蒙混过关。
4. **为覆盖率写无断言测试**：数字漂亮，实则没验证任何行为。
5. **flaky 长期隔离无人修**：隔离区变垃圾堆，真问题被埋。
6. **全量慢测试塞进每个 PR**：反馈 40 分钟，没人愿等，干脆跳过。
7. **CI 与本地命令不一致**：本地绿 CI 红，无法复现。

## 8. 最低交付 checklist

- [ ] 分支保护开启，关键检查设为 required，红则不可合。
- [ ] PR 门禁 < 10 分钟，包含 lint/类型/单测/集成/契约/构建/安全扫描。
- [ ] 慢测试（全 E2E/性能/变异/可视回归）放夜间，不拖 PR。
- [ ] 变更覆盖率达阈值 + 总覆盖率不下降，核心模块要分支覆盖。
- [ ] flaky 有检测、隔离、限期修流程，不靠"重跑"掩盖。
- [ ] 测试数据隔离、可并行、确定性（固定种子/时钟）。
- [ ] CI 跑的命令与本地一致、可复现。
