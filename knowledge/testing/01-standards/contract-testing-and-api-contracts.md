# 契约测试与接口契约规范（商业级必读）

> 前后端、服务与服务之间靠"接口契约"协作。只跑各自的单测，无法发现"后端悄悄改了字段、前端还在用旧字段"这类**集成断裂**。
> 契约测试用机器可验证的方式锁住接口形状：一方改了不兼容的契约，CI 立刻红，而不是等到联调或生产才炸。

## 1. 契约测试解决什么

- **集成断裂在合并前暴露**：消费方（前端/下游服务）对提供方（API/上游服务）的期望被固化成可执行契约；提供方改坏了，提供方的 CI 就失败。
- **不必拉起整套环境**：契约测试用契约 + 桩，比端到端联调快得多、稳得多，可在每次 PR 跑。
- **明确的责任边界**：契约是双方共同签署的事实来源；谁破坏谁负责。

契约测试**不替代**集成/E2E，它补在金字塔的"集成层"，专门保接口对齐（见 `testing/01-standards/test-strategy-and-layering`）。

## 2. 两种契约校验

### A. 模式契约（Schema Contract）
以接口定义文件（OpenAPI / GraphQL SDL / protobuf）为单一事实来源，校验**真实流量**符合模式：
- **提供方**：集成测试里，断言每个端点的真实响应满足声明的模式（字段、类型、必填、状态码）。响应多/少/错字段即失败。
- **消费方**：从前端代码里抽取所有 `fetch`/HTTP 调用，断言每个被调用的"路径+方法+请求/响应形状"都存在于契约中——杜绝"调了一个契约里不存在的端点"或"端点在活跃消费方下被悄悄改形状"。
- **破坏性 diff**：把新契约与上一发布版对比，删字段/改类型/收紧校验/改状态码语义=破坏性变更，未升版本则阻断合并。

### B. 消费者驱动契约（Consumer-Driven Contract）
当多个消费方依赖同一提供方时，让**消费方**声明"我实际用到的请求/响应子集"，汇总成提供方必须满足的契约：
```
1. 每个消费方写出它对提供方的期望（请求样例 → 期望响应样例）。
2. 这些期望被收集（共享契约仓/broker），形成提供方的契约集合。
3. 提供方在 CI 回放所有消费方期望：任一不满足即失败。
4. 部署前用"能否部署(can-i-deploy)"判定：该版本是否兼容当前所有线上消费方。
```
价值：提供方只需保证"被真正用到的部分"稳定，敢于删改无人使用的字段；消费方不会被上游静默破坏。

## 3. 工作流

```
契约先行（见 experts/architect/contract-first-api-design）
      │
      ├─ 提供方：实现 → 契约校验测试（真实响应 vs 模式）→ 回放消费方期望
      │
      └─ 消费方：对着契约/Mock 开发 → 用契约桩做集成测试（不打真实提供方）
                                  │
                          CI 门禁：契约 lint + 双向校验 + 破坏性 diff + can-i-deploy
```

- 消费方测试用**契约生成的 Mock/桩**代替真实提供方，既快又能在提供方未就绪时并行开发。
- 提供方测试用**真实实现**回放契约，确保不是"对着自己的假设测自己"。

## 4. 契约必须锁住的内容

| 维度 | 校验点 |
|---|---|
| 路径 / 方法 | 每个被消费的端点都存在 |
| 请求 | 必填/可选字段、类型、格式、枚举、范围 |
| 响应（按状态码）| 成功与各错误码的 body 形状 + 示例 |
| 状态码集合 | 200/201/204/400/401/403/404/409/422/429/5xx 按操作声明 |
| 错误信封 | 全局统一的错误结构（code/message/details） |
| 鉴权 | 哪些端点需鉴权、需要的 scope/role |
| 分页 / 幂等 | 集合的分页契约、不安全操作的幂等键 |
| 版本 | 破坏性变更的信号与版本策略 |

## 5. 兼容性规则

- **加字段、加端点、放宽校验 = 兼容**（消费方须容忍未知字段——收到多余字段不报错）。
- **删/改字段、改类型、收紧校验、改状态码语义 = 破坏**：必须升主版本 + 走弃用流程（标记弃用 → 公告 → sunset 日期 → 旧版本流量耗尽后再删）。
- 把"契约破坏性 diff"做成 CI 检查：未升版本的破坏性变更直接 fail。

## 6. CI 门禁（契约相关）

每次 PR 应执行：
- [ ] **契约 lint**：模式合法、命名一致、每个操作带示例。
- [ ] **提供方响应校验**：集成测试断言真实响应符合契约。
- [ ] **消费方调用校验**：前端/下游调用的路径与形状都在契约内。
- [ ] **消费者期望回放**：提供方满足所有已注册消费方契约。
- [ ] **破坏性 diff**：对比上一发布契约，破坏未升版即阻断。
- [ ] **can-i-deploy**：发布前确认该版本兼容当前线上各消费方。

## 7. 反模式（出现即不合格）

1. **只有单测、无契约校验**：前后端各自绿，集成处断裂。
2. **契约是写完代码后补的文档**：从未被任何 CI 检查，与代码漂移。
3. **消费方对着真实提供方做集成**：慢、脆、提供方未就绪就卡住；该用契约桩。
4. **提供方对着自己的假设测自己**：没回放真实消费方期望。
5. **静默破坏性变更**：改字段不升版本，线上消费方被打挂。
6. **不容忍未知字段**：提供方加个可选字段，消费方就崩——脆契约。

## 8. 最低交付 checklist

- [ ] 接口有机器可读契约（OpenAPI/SDL/proto）作为单一事实来源。
- [ ] 提供方有"真实响应 vs 契约"的自动化校验测试。
- [ ] 消费方有"调用 vs 契约"的自动化校验，且用契约桩做集成。
- [ ] 多消费方场景启用消费者驱动契约 + can-i-deploy。
- [ ] 破坏性变更检测进 CI，未升版本即阻断合并。
- [ ] 弃用走"标记→公告→sunset→下线"流程，不静默删改。
