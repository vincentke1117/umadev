# 测试数据管理与临时环境规范（商业级必读）

> 测试不可信，往往不是因为代码错，而是因为**数据脏、环境脏**：上一条用例残留的数据污染下一条、用生产真实个人数据跑测试、几十个分支抢同一个共享 staging。
> 商业级团队把"测试数据如何造、如何隔离、用完如何清"和"每个改动跑在自己的一次性环境里"当成工程能力来建，而不是靠人手摆数据。

## 1. 核心原则

- **确定性**：同一套用例每次跑的数据状态一致，结果可复现；不依赖"今天库里恰好有什么"。
- **隔离**：每个用例/每条流水线/每个分支拥有自己的数据与环境，互不串味，可并行。
- **自给自足**：用例自己准备所需数据（前置），不假设环境里已经存在某条记录。
- **用完即清**：数据和环境都有明确的销毁路径，绝不无限堆积、绝不长期占用。
- **绝不用真实个人数据**：测试库里不放生产 PII，用脱敏或合成数据（见 `compliance/01-standards/privacy-and-compliance-readiness`）。

## 2. 测试数据的四种来源

| 方式 | 做法 | 适用 | 注意 |
|---|---|---|---|
| 工厂/构建器（factory）| 用代码按需生成对象，字段给合理默认、按用例覆盖关键字段 | 单元/集成测试首选 | 默认值要合法，只显式写"与本用例相关"的字段 |
| 固定夹具（fixture）| 预定义一组静态数据集，测试前加载 | 参照数据、枚举、种子账号 | 易腐化，变更要同步；别塞进巨型共享夹具 |
| 合成数据（synthetic）| 按规则/分布批量生成仿真数据 | 性能/容量/边界测试 | 要覆盖边界与异常分布，不只是"理想数据" |
| 脱敏生产数据 | 取生产快照后**不可逆脱敏/掩码**再用 | 需要真实形态/规模时 | 脱敏必须在数据离开生产边界前完成；保留引用完整性 |

要点：优先工厂法（表达力强、随用例演进）；固定夹具只放稳定的参照数据；**绝不**直接拷生产库进测试环境。

## 3. 数据脱敏与合成

- **脱敏规则**：姓名/邮箱/手机/证件/卡号等可识别字段一律替换、掩码或令牌化；保留格式有效性（邮箱仍是邮箱）和**引用完整性**（外键仍能对上）。
- **不可逆**：脱敏后不能反推回真人——别用可逆加密冒充脱敏。
- **一致性**：同一真实值在多表脱敏成同一假值，避免 join 断裂。
- **合成优先**：能用规则造的就别取生产；合成数据零隐私风险、可覆盖边界，是长期方向。

## 4. 测试数据生命周期（每条用例必须闭环）

```
准备(setup) → 执行(act) → 断言(assert) → 清理(teardown)
```

- **准备**：用例自己创建所需数据，带唯一标识（如随机后缀）避免与并发用例撞键。
- **清理**：优先用**事务回滚**（用例跑在事务里，结束回滚，最快最干净）；不能回滚时用显式删除或在隔离的库/schema 里跑。
- **不依赖执行顺序**：用例之间无隐式依赖，任意打乱/单独跑都通过。
- **不留垃圾**：失败的用例也要能清理（teardown 放在 finally/fixture 析构里）。

## 5. 隔离级别（从强到弱）

| 级别 | 隔离手段 | 并行度 | 成本 |
|---|---|---|---|
| 进程内 | 内存库/事务回滚 | 高 | 低 |
| schema/库级 | 每个 worker 一套 schema 或独立库 | 高 | 中 |
| 容器级 | 每条流水线起一套临时容器栈 | 中 | 中 |
| 环境级 | 每个分支/PR 一套临时环境 | 中 | 高 |

原则：能用更轻的隔离就别上更重的；CI 里并行跑的测试**必须**至少做到 schema/库级隔离，否则互相污染、随机翻红。

## 6. 临时/预览环境（ephemeral / preview）

- **定义**：为一次改动（一个 PR/一个分支）**自动拉起、用完自动销毁**的完整运行环境，让评审者和 QA 能在合并前看到真实运行效果。
- **声明式 + 自动化**：环境由代码/配置定义（基础设施即代码），从零一键重建；**绝不**手工攒环境。
- **自动生命周期**：PR 打开 → 自动部署预览环境 → 评论里回链接；PR 合并/关闭 → 自动销毁，绝不遗留。
- **种子数据**：预览环境用脱敏/合成种子数据自动初始化，达到可演示状态，不连生产数据。
- **隔离与成本**：每个预览环境互相隔离；设**TTL 自动回收**和资源上限，防止环境无限堆积烧钱。
- **与渐进式发布衔接**：预览环境用于"合并前看效果"，与上线后的金丝雀放量是两件事（见 `release-engineering/01-standards/progressive-delivery-and-release`）。

## 7. 环境分层与数据策略

| 环境 | 数据 | 持久性 |
|---|---|---|
| 本地/单测 | 工厂生成 + 事务回滚 | 即用即弃 |
| CI 集成测试 | 容器内临时库 + 种子 | 流水线结束即销毁 |
| 预览（每 PR）| 脱敏/合成种子 | PR 关闭即销毁 |
| staging | 接近生产的脱敏数据 | 长期，但定期刷新与清理 |
| 生产 | 真实数据 | 受合规与保留策略约束 |

staging 不是垃圾场：定期刷新脱敏数据、清理过期记录，让它真正"接近生产"。

## 8. 反模式（出现即不合格）

1. **测试间共享可变数据**：用例 A 造的数据被用例 B 读，顺序一变就翻红。
2. **生产数据直灌测试**：把真实 PII 拷进测试/预览环境，隐私事故。
3. **不清理**：用例只建不删，库越跑越脏，最后只能"重建环境"。
4. **巨型共享夹具**：一个庞大夹具喂所有用例，改一处崩一片，没人敢动。
5. **手工长寿 staging**：靠人手摆的"祖传环境"，没人能从零重建。
6. **抢占共享环境**：几十个分支共用一个 staging，互相覆盖、排队等待。
7. **预览环境不回收**：临时环境只建不销，资源无限堆积。
8. **依赖隐式现存数据**：用例假设"库里本来就有 id=1 的用户"。

## 9. 最低交付 checklist

- [ ] 测试数据用工厂/构建器按需生成，带唯一标识，不撞键。
- [ ] 每条用例自给自足、不依赖执行顺序、用完即清（优先事务回滚）。
- [ ] CI 并行测试至少做到 schema/库级隔离，互不污染。
- [ ] 测试/预览环境绝不使用生产真实个人数据，统一脱敏或合成。
- [ ] 脱敏不可逆、保持格式有效与引用完整、跨表一致。
- [ ] 预览环境由 IaC 定义，PR 打开自动拉起、关闭自动销毁，带 TTL 回收。
- [ ] 预览环境自动种子数据到可演示状态。
- [ ] staging 数据定期刷新与清理，保持接近生产。
